작게 배포하고 빠르게 배우는 블로그 시작기

Jenkins와 GitHub Pages를 묶을 때 처음부터 크게 설계하지 않고, 작은 배포 루프로 시작하는 이유를 정리했습니다.

Woon Yong 2026년 3월 14일 · 1분 읽기 · 수정 2026년 3월 31일
Jenkins GitHub Pages Deployment

블로그를 열 때 가장 먼저 본 것은 기술 스택보다 끊기지 않는 배포 흐름이었습니다.

새 글을 올릴 때마다 빌드 방법을 다시 떠올려야 하거나, 실패했을 때 복구 경로가 불분명하면 결국 글 쓰는 리듬이 깨집니다. 그래서 시작 단계에서는 기능을 많이 넣기보다 아래 세 가지를 먼저 고정하는 편이 좋습니다.

1. 결과물은 항상 정적 파일로 남긴다

Jenkins가 하든 GitHub Actions가 하든, 최종 산출물이 단순한 HTML과 CSS, JS라면 복구가 훨씬 쉽습니다.

  • 어떤 브랜치에 배포되는지 명확하다
  • 실패했을 때 이전 커밋으로 되돌리기 쉽다
  • 호스팅 대상을 바꾸더라도 옮기기 쉽다

이번 블로그도 같은 이유로 gh-pages 브랜치에 빌드 결과만 올리는 구조를 선택했습니다.

2. 배포 파이프라인은 한 번에 많은 걸 하지 않는다

처음부터 이미지 최적화, 검색 인덱싱, 다국어, 댓글, 분석 도구를 전부 넣으면 실패 지점이 늘어납니다.

저는 보통 파이프라인을 아래 순서로 키웁니다.

설치 -> 빌드 -> 산출물 검증 -> 배포

이 네 단계만 분리되어 있어도 어디에서 실패했는지 바로 보입니다.

3. 디자인 수정도 배포 루프 안에서 본다

블로그 테마는 로컬에서만 보면 생각보다 자주 착시가 생깁니다. 프로젝트 페이지 경로, 실제 글 길이, 모바일 폭, 다크 모드 토큰까지 같이 봐야 합니다.

그래서 테마를 바꿀 때도 “완성본을 한 번에 만든다”보다 “작은 수정 후 바로 빌드해서 본다”가 더 잘 맞았습니다.

마무리

블로그를 오래 운영하려면 멋진 첫 화면보다 다음 글을 쉽게 올릴 수 있는 구조가 먼저 필요합니다. 작게 배포하고 빠르게 확인할 수 있는 루프가 있으면 디자인도, 글도, 구조도 훨씬 덜 지칩니다.
