왜 Astro였나

개발 블로그는 인터랙션보다 읽기 경험과 관리 비용이 더 중요했습니다. 그래서 클라이언트 상태 관리보다 정적 생성이 편한 쪽을 고르는 게 맞았습니다.

Astro는 이런 요구에 잘 맞습니다.

  • 마크다운 기반 글 작성이 편합니다.
  • 빌드 결과물이 단순해서 Jenkins와 궁합이 좋습니다.
  • 페이지 성능을 챙기기 쉽습니다.

Jenkins를 남겨둔 이유

GitHub Actions로도 충분히 배포할 수 있지만, 이미 운영 중인 Jenkins가 있다면 블로그도 같은 파이프라인에 올려두는 편이 관리에 유리합니다.

npm ci
npm run build
git push origin gh-pages

핵심은 복잡한 자동화보다 실패 지점이 적은 흐름을 만드는 것입니다.

초기에 넣고 싶은 기준

처음부터 완벽한 운영 글을 쓸 필요는 없습니다. 대신 아래 기준은 초반부터 잡아두면 좋았습니다.

  • 제목만 보고도 글의 문제의식이 보일 것
  • 10분 뒤 다시 읽어도 이해되는 문장으로 쓸 것
  • 설정 메모와 회고를 분리하지 말 것

마무리

이 글은 실제 운영 콘텐츠가 아니라, 블로그 초기 구성을 테스트하기 위한 더미 포스트입니다. 그래도 이후에 진짜 글로 확장하기 쉬운 톤과 구조를 의도했습니다.
